# Driver Dashboard Implementation Plan

## Goal
Create a fully functional **Driver Dashboard** similar in structure and UX to the existing **Store Dashboard** (`frontend/src/pages/store_dashboard/StoreDashboard.tsx`), leveraging the existing `DriverModel`, `DriverController`, and `DriverSignUpPage`.

## Current State Analysis
- **Driver backend ready**: Full schema (status, live toggle, vehicle, zones, documents, earnings, ratings, operationTimes).
- **Controller endpoints**: `createDriver`, `getDriverProfile`, `updateDriverProfile`, `uploadDriverDocuments`.
- **No frontend dashboard yet**: Only signup page exists.
- **Store dashboard pattern** (reference):
  - `StoreDashboard.tsx` handles auth, loading, team checks, data fetching, sidebar + topbar, nested Routes.
  - Supporting pages under `supporting_pages/`: overview, orders, products/services/etc, settings.
  - Reusable components: stat cards, tables, filters, modals, menubar/topbar.
- **Main dashboard** (`pages/dashboard/Dashboard.tsx`): Simple menubar + topbar + routes pattern.

## Proposed Structure
- New folder: `frontend/src/pages/driver_dashboard/`
  - `DriverDashboard.tsx` (main shell, mirroring StoreDashboard)
  - `supporting_pages/`:
    - `DriverDashboardOverview.tsx` (live toggle, stats, quick actions)
    - `DriverJobs.tsx` (current/available deliveries, orders table)
    - `DriverEarnings.tsx` (earnings breakdown, payout history)
    - `DriverZones.tsx` (collection/delivery zone management)
    - `DriverProfile.tsx` (vehicle/docs/operation times)
    - `DriverSettings.tsx` (account settings, availability)
  - `components/` (driver-specific or shared reuse):
    - Menubar: `DriverDashboardMenubar.tsx`
    - Topbar: `DriverDashboardTopbar.tsx`
    - Cards: `DriverStatCard.tsx`, `LiveStatusToggle.tsx`
    - Tables: `DriverJobsTable.tsx`
    - Filters: `DashboardFilterByStatus.tsx` (reuse/adapt)

## Key Features (MVP)
1. **Live/Availability Toggle** — `isLive` + `lastSeenAt` from model.
2. **Jobs/Deliveries** — List assigned + available orders (requires new backend endpoints).
3. **Earnings & Payouts** — Use `earnings` schema (totalEarned, pendingPayout, totalDeliveries).
4. **Zones Management** — CRUD for `collectionZones` / `deliveryZones` (GeoJSON).
5. **Profile & Documents** — View/update vehicle, re-upload docs, operationTimes.
6. **Stats** — Rating, deliveries count, performance insights (adapt StoreDashboardStatCardsBoard).

## Backend Additions Needed
- New controller methods/routes (in `DriverController.js` + router):
  - `getDriverDashboardSummary` (earnings + stats)
  - `getAvailableJobs` / `getMyJobs`
  - `updateZones`
  - `toggleLiveStatus` (use existing `toggleLive` instance method)
- Add to `backend/server.js` under protected driver routes.
- Ensure auth middleware checks `driver.accountStatus === 'active'`.

## Routing
- Add route in main router (similar to `/dashboard/:storeSlug`):
  - `/driver-dashboard` → `DriverDashboard.tsx`
- Protect with existing `ProtectedRoute` + driver role check.
- Update `authRoutes.tsx` or main app routes if needed.

## Implementation Steps
1. Create `DriverDashboard.tsx` shell (sidebar + topbar + Routes + auth guard using driver profile).
2. Build supporting overview page with live toggle + stat cards.
3. Implement jobs/earnings pages + reusable table components.
4. Add zone management UI (map or list-based, reuse existing geo patterns).
5. Wire data fetching via new Redux slices or direct API calls (mirror `storeAdminSlice`).
6. Add backend endpoints first for data.
7. Style consistently with Tailwind + existing slate/purple theme.
8. Add loading/error states, mobile menu handling (copy from StoreDashboard).

## Files to Create/Modify

  - All supporting_pages + menubar/topbar components.

  - Main routes file (likely `frontend/src/App.tsx` or routes index).

## Tradeoffs & Clarifications Needed
- Should driver dashboard live under `/dashboard/driver` or dedicated `/driver-dashboard`?
- Map integration for zones (Leaflet/Google Maps already in project?) or simple list UI first?
- Order matching logic — will backend provide "available jobs" endpoint soon, or mock for now?
- Reuse existing store dashboard components heavily or create parallel driver set?

This plan provides a complete, reference-based roadmap ready for implementation.
